Blocking synchronous calls should have associated timeouts

Technology

JEE

Unified Status

N/A

XXL Status

N/A

Critical Quality Rule

No

Weight in Parent Tech. Criterion

9

Rationale

Blocking calls can result in system failure if the called process fails. If the timeout is infinite that may cause the application to hang forever. Operations on the standard input/output streams depend directly on other processes outside of the Java VM. If they decide to block forever, so will reads/writes to these streams in the application.

Description

This quality rule reports synchronous call that don't use timeout to prevent an infinite waiting time. Here is the list of control that are performed: * Socket - java.net.Socket.connect(SocketAddress endpoint) or connect(SocketAddress endpoint, int 0)/ remediation connect(SocketAddress endpoint, int timeout) - true for child classes too * Future: a Future represents the result of an asynchronous computation. - java.util.concurrent.Future<V>.get() / remediation use java.util.concurrent.Future<V>.get(long timeout, TimeUnit unit) instead for all classes that implement this interface such as java.util.concurrent.FutureTask<V>, java.util.concurrent.ForkJoinTask<V> ... and all classes that implement a sub-interface of Future<V>. * JMS: - all call to methods that implement javax.jms.MessageConsumer.receive() or javax.jms.MessageConsumer.receive(0) / remediation call javax.jms.MessageConsumer.receive(long timeout)
